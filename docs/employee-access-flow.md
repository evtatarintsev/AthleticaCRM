# Флоу добавления сотрудника и выдачи доступа

## Обзор

Администратор добавляет сотрудника вручную через интерфейс. После добавления сотрудник не может войти в систему — доступ выдаётся отдельным действием: администратор задаёт пароль и отправляет его на email сотрудника.

---

## Шаги

### 1. Администратор создаёт сотрудника

- Открывает раздел **Сотрудники** → нажимает **Добавить сотрудника**.
- Заполняет форму: имя, **email** (обязателен), телефон (опционально), фото (опционально).
- Нажимает **Создать**.

**Что происходит на сервере (`POST /api/employees/create`):**

1. Валидация: `email` обязателен — без него создание невозможно.
2. В одной транзакции атомарно создаются две записи:
   - `users`: `login = email`, `password_hash` — случайный заблокированный хеш (сотрудник не может войти).
   - `employees`: `is_active = false` — аккаунт неактивен.
3. Возвращается созданный сотрудник со статусом **Неактивен**.

> Пока `employees.is_active = false`, вход в систему для этого аккаунта невозможен — проверка выполняется на уровне запроса аутентификации.

---

### 2. Администратор отправляет доступ

- В списке сотрудников выбирает **одного неактивного** сотрудника (чекбокс).
- В нижней панели действий нажимает иконку **Отправить доступ** (активна только при выборе одного неактивного сотрудника).
- В диалоге вводит пароль для сотрудника и подтверждает.

**Что происходит на сервере (`POST /api/employees/send-access`):**

1. Находит сотрудника в организации по `employeeId`.
2. В одной транзакции атомарно:
   - Обновляет `users.password_hash` на хеш введённого пароля.
   - Устанавливает `employees.is_active = true`.
3. Отправляет email сотруднику с логином (email) и паролем.
4. Фиксирует событие `employee_access_sent` в журнале аудита.

---

### 3. Сотрудник входит в систему

- Получает письмо с логином (свой email) и паролем.
- Входит через стандартную форму авторизации.
- При желании меняет пароль в **Настройки → Сменить пароль**.

---

### 4. Восстановление доступа

Если сотрудник потерял доступ или забыл пароль — администратор повторяет **шаг 2**:
выбирает сотрудника, вводит новый пароль и отправляет. Новый пароль немедленно становится действующим, `is_active` остаётся `true`.

---

## Возможные ошибки при создании

| Код | Причина |
|-----|---------|
| `EMPLOYEE_EMAIL_REQUIRED` | Email не указан в запросе |
| `EMPLOYEE_EMAIL_IN_USE` | Пользователь с таким email уже зарегистрирован |
| `EMPLOYEE_ALREADY_EXISTS` | Сотрудник с переданным `id` уже существует |

## Возможные ошибки при отправке доступа

| Код | Причина |
|-----|---------|
| `EMPLOYEE_NOT_FOUND` | Сотрудник не найден в организации |
