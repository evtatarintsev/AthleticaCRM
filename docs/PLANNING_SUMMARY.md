## 🎯 Рекомендуемый MVP (Minimum Viable Product)

### Phase 1: Базовая система работает (Спринты 1-2, ~4 недели)

**Основная идея:** Администратор может продать абонемент клиенту и отметить посещение.

**Что нужно реализовать:**

1. **Планы абонементов (MembershipPlan)**
   - UC-2.1: CreateMembershipPlan, UpdateMembershipPlan, ListMembershipPlans, ArchiveMembershipPlan
   - Типы: классический (X занятий), безлимитный, день-в-день, пакет
   - API: CRUD операции

2. **Абонементы (Membership)**
   - UC-2.3: CreateMembership (продажа), ListClientMemberships, MembershipDetail
   - UC-2.5: RecordPayment (регистрация оплаты)
   - Состояния: pending_payment → active → expired/exhausted
   - Автоматический расчёт expires_at

3. **Занятия (Session)**
   - UC-4.2: GenerateSessions (генерирование из расписания группы)
   - UC-4.2: AssignTrainerToSession (назначение тренера)
   - Просмотр расписания на день/неделю

4. **Посещаемость (Attendance)**
   - UC-3.1: CheckinClient (отметить посещение)
   - Автоматическое убыль занятия из абонемента (DeductMembership)
   - UC-3.1.4: CancelCheckin (отмена посещения, восстановление занятия)

5. **Залы (Room)**
   - UC-4.3: CreateRoom, UpdateRoom, ListRooms
   - UC-4.3: AssignGroupToRoom
   - Проверка конфликтов расписания

6. **Расширение клиентской базы**
   - UC-1.2: UpdateClientInfo, UpdateClientContacts, AddClientNote
   - UC-1.3: ClientActivityTimeline (объединённая история)

### Примерный объём
- **Моделей данных:** 6-8 новых таблиц (MembershipPlan, Membership, Session, Attendance, Room, и связи)
- **Юзкейсов:** ~20-25 new use case functions
- **API endpoints:** ~30-35 новых endpoint-ов
- **Тестов:** unit + integration для критических путей (A, B, C, D)

### Что НЕ входит в Phase 1
- ❌ Интеграция с платежными системами (только ручная регистрация платежа)
- ❌ Отправка уведомлений (SMS, email) — инфраструктура готова, но логика нет
- ❌ Аналитика и отчеты (Phase 2)
- ❌ Экспорт в календари (Phase 2)
- ❌ Льготы и сезонные планы абонементов (Phase 2)

## ⚠️ Критичные решения, которые нужно принять

### 1. Что такое "группа" в системе?
**Текущее понимание:** Группа = курс (например, "Йога уровень 1" по ПН/СР/ПТ)

**Нужно уточнить:**
- Одна группа может собираться в разных залах?
- Одна группа может быть у разных тренеров?
- Может ли быть "группа с ротацией тренеров"?

**Рекомендация:** Сейчас привязываем Group → One Room, One Trainer (можно менять). В Phase 2 расширим.

### 2. Как работает день-в-день (drop-in)?
**Текущее понимание:** Клиент покупает абонемент "день-в-день" и может посетить любую группу один раз.

**Нужно уточнить:**
- Списание денег за drop-in происходит при посещении или при покупке?
- Может ли drop-in клиент посетить occupancy-ограниченное занятие (если зал переполнен)?

**Рекомендация:** Drop-in = абонемент с 1 занятием, требует явной оплаты перед посещением. Можно занять место если есть свободно.

### 3. Возврат абонемента (refund)?
**Текущее понимание:** Пока не поддерживаем.

**Нужно уточнить:**
- Может ли клиент вернуть неиспользованный абонемент?
- На какой день? (30 дней на раздумье?)
- Полный возврат или минус комиссия?

**Рекомендация:** Phase 1 — не поддерживаем возвраты. Phase 2 — добавим с финансовой логикой.

### 4. Переносимость занятий?
**Сценарий:** Клиент был записан на ПН 18:00, но не пришел. Может ли он посетить СР 18:00 в тот же день/неделю?

**Рекомендация:** 
- Phase 1: Нет, только отмена и возврат занятия в абонемент
- Phase 2: Да, TransferCheckin с правилами (не позже N дней, только в ту же дисциплину)

## 📐 Архитектурные выводы

### Слой данных (Database)
- PostgreSQL с R2dbc — хороший выбор для размера ≤1000 клиентов
- Нужны индексы на org_id, status, dates
- Transactions для критичных операций (Membership deduction)
- JSON для неструктурированных данных (Notes, Metadata)

### Слой бизнес-логики
- Использовать Arrow Either/Result для обработки ошибок ✅
- Валидировать на входе (request schemas)
- Использовать domain-specific errors (не generic exceptions)
- Логировать всё в audit_log для compliance

### Слой API
- Использовать OpenAPI/Swagger документацию
- Версионировать API (/v1/api)
- Вернуть правильные HTTP коды (201 для Create, 400 для validation errors)
- Возвращать структурированные ошибки (code + message)

### Состояния и переходы
- Использовать enum для status полей (не строки)
- Документировать state machines (какие переходы разрешены)
- Добавить логирование при смене status

## 🧪 Стратегия тестирования

### Unit Tests (быстрые)
- Валидация данных
- Бизнес-логика (e.g., calcExpireDate)
- Обработка ошибок

### Integration Tests (медленнее)
- Критичные потоки: A (покупка), B (посещение), C (отмена), D (истечение)
- Проверка конфликтов занятий в залах
- Транзакции

### E2E Tests (опционально для MVP)
- Full user journey: создать клиента → купить абонемент → посетить занятие

### Данные для тестирования
- Fixture: несколько организаций, групп, планов абонементов, клиентов
- Seed database скрипт для локальной разработки
