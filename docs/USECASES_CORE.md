## 1. Клиентская база

### Статус реализации
- ✅ **CreateClient** — создание клиента
- ✅ **ClientList** — список клиентов  
- ✅ **ClientDetail** — детали клиента
- ✅ **AdjustClientBalance** — редактирование баланса
- ✅ **ClientBalanceHistory** — история платежей
- ✅ **AddClientsToGroup** / **RemoveClientsFromGroup** — управление участием в группах
- ⏳ **Нужно реализовать** (см. ниже)

### Юзкейсы для реализации

#### 1.1 Просмотр и фильтрация клиентов (расширение)
**Актеры:** Администратор, рецепционист

**Кейсы:**
- UC-1.1.1: Просмотр списка всех клиентов организации с сортировкой (по имени, дате регистрации, балансу)
- UC-1.1.2: Поиск клиента по имени/контактам
- UC-1.1.3: Фильтрация клиентов по статусу (активный, неактивный, заблокирован)
- UC-1.1.4: Фильтрация по времени последнего визита
- UC-1.1.5: Экспорт списка клиентов (CSV, XLSX)

**Статус:** CreateClient, ClientList, ClientDetail уже есть. Нужны:
- расширенные фильтры в ClientList
- SearchClients (быстрый поиск)
- ExportClients
- ClientStatistics (статус, последний визит)

#### 1.2 Управление профилем клиента
**Актеры:** Администратор, рецепционист, сам клиент (ограниченно)

**Кейсы:**
- UC-1.2.1: Редактирование основной информации (имя, контакты, дата рождения, пол)
- UC-1.2.2: Загрузка/смена аватара
- UC-1.2.3: Добавление дополнительных контактов (несколько номеров/email)
- UC-1.2.4: Добавление заметок о клиенте (внутренние комментарии)
- UC-1.2.5: Пометка клиента как VIP
- UC-1.2.6: Блокировка клиента (например, за непогашенный долг)

**Статус:** CreateClient есть. Нужны:
- UpdateClientInfo
- UpdateClientContacts
- UpdateClientAvatar
- AddClientNote / UpdateClientNote / DeleteClientNote
- MarkClientAsVIP
- BlockClient

#### 1.3 История активности клиента
**Актеры:** Администратор

**Кейсы:**
- UC-1.3.1: Просмотр временной шкалы активности (посещения, платежи, изменения в абонементе)
- UC-1.3.2: Просмотр истории изменения данных клиента (кем, когда, что изменилось)
- UC-1.3.3: Просмотр истории всех платежей/операций по балансу

**Статус:** ClientBalanceHistory есть. Нужны:
- ClientActivityTimeline (объединённая история)
- ClientDataChangeHistory (audit для основных полей)

#### 1.4 Управление статусами и сегментацией
**Актеры:** Администратор, маркетолог (когда появится)

**Кейсы:**
- UC-1.4.1: Добавление клиента в группу/тег (для сегментации)
- UC-1.4.2: Удаление клиента из группы/тега  
- UC-1.4.3: Просмотр всех групп/тегов клиента
- UC-1.4.4: Массовое добавление клиентов в группу

**Статус:** AddClientsToGroup / RemoveClientsFromGroup есть.
- Уточнить: что такое "группа" — это группа занятий или сегмент клиентов? Нужна ясность.

#### 1.5 Уведомление о удалении клиента (безопасность)
**Актеры:** Администратор

**Кейсы:**

- UC-1.5.3: Восстановление архивированного клиента

## 2. Абонементы & тарификация

### Статус реализации
- ❌ Полностью отсутствует

### Юзкейсы

#### 2.1 Управление тарифными планами
**Актеры:** Администратор

**Кейсы:**
- UC-2.1.1: Создание нового тарифного плана (название, описание, цена, период действия)
- UC-2.1.2: Типы планов:
  - **Классический абонемент**: Х занятий в период (месяц/квартал/год)
  - **Безлимитный**: Неограниченное кол-во занятий в период
  - **День-в-день** (drop-in): Одно занятие без накопления
  - **Пакет занятий**: Х занятий без срока (не привязано к календарю)
- UC-2.1.3: Редактирование плана (только если на него ещё никто не подписан)
- UC-2.1.4: Архивирование плана (выключение продаж, но сохранение для активных подписок)
- UC-2.1.5: Просмотр списка всех планов

**Статус:** Нужно реализовать:
- CreateTariff / UpdateTariff / ArchiveTariff
- ListTariffs
- Модели данных для разных типов планов

#### 2.2 Расширенная тарификация (опционально для 1.0, но важно спланировать)
**Актеры:** Администратор

**Кейсы:**
- UC-2.2.1: Привязка плана к конкретной дисциплине/группе (разные цены для разных курсов)
- UC-2.2.2: Сезонные тарифы (летний лагерь дороже обычного)
- UC-2.2.3: Скидки для новичков (первый месяц 50%)
- UC-2.2.4: Льготы для постоянных клиентов (каждый Nй месяц дешевле)

**Статус:** Сложнее. Для MVP это может быть в Phase 2.

#### 2.3 Управление абонементами клиента
**Актеры:** Администратор, рецепционист

**Кейсы:**
- UC-2.3.1: Продажа абонемента клиенту (выбор плана → генерация абонемента)
- UC-2.3.2: Просмотр активных абонементов клиента
- UC-2.3.3: Просмотр истории абонементов (закончившихся, аннулированных)
- UC-2.3.4: Продление абонемента (автоматическое или ручное)
- UC-2.3.5: Отмена/возврат абонемента (полностью или частично)
- UC-2.3.6: Переоформление абонемента на другого клиента (транспортировка)
- UC-2.3.7: Передача абонемента членам семьи (если организация поддерживает)

**Статус:** Основное, нужно реализовать:
- CreateMembership (продажа)
- ListClientMemberships (активные)
- MembershipDetail (детали, остаток занятий)
- RenewMembership
- CancelMembership
- TransferMembership

#### 2.4 Управление занятиями в абонементе
**Актеры:** Администратор, рецепционист (учет)

⚠️ **ВАЖНО:** Списание происходит при MarkInstanceAsCompleted, а не при CheckinClient (см. паттерн в [`ATTENDANCE_DEDUCTION_PATTERNS.md`](./ATTENDANCE_DEDUCTION_PATTERNS.md))

**Кейсы:**
- UC-2.4.1: Убыль занятия из абонемента (при завершении занятия)
  - Происходит в MarkInstanceAsCompleted (АТОМАРНО для всех посещений)
  - ✅ Все в одной транзакции
  
- UC-2.4.2: Восстановление занятия (если клиент отменил визит)
  - Если черновик (REGISTERED): просто RemoveAttendance (баланс не менялся)
  - Если финал (ATTENDED): CancelAttendance (восстановить баланс)
  
- UC-2.4.3: Просмотр остатка занятий в активных абонементах
  - balance = кол-во оставшихся занятий
  
- UC-2.4.4: Уведомление при низком остатке (например, ≤2 занятия)
  - После MarkInstanceAsCompleted проверяем balance
  - Если ≤2 → отправить уведомление
  
- UC-2.4.5: Истечение срока абонемента (архивирование, уведомление)
  - Фоновый процесс ежедневно

**Статус:** Тесно связано с посещаемостью. Нужно:
- DeductMembership (убыль в MarkInstanceAsCompleted)
- RestoreMembership (восстановление при CancelAttendance)
- GetMembershipBalance (остаток)
- CheckMembershipLowBalance (уведомление)
- Фоновый процесс для истечения сроков

#### 2.5 Платежи за абонементы (базовое)
**Актеры:** Администратор, клиент (при оплате сам)

**Кейсы:**
- UC-2.5.1: Регистрация оплаты абонемента (наличные, карта, банк)
- UC-2.5.2: Просмотр неоплаченных абонементов
- UC-2.5.3: Напоминание об оплате (если абонемент куплен но не оплачен)
- UC-2.5.4: Выписка по платежам (для финансового учета)

**Статус:** Базовое, нужно:
- RecordPayment
- ListUnpaidMemberships
- PaymentReceipt (генерирование)

Интеграция с платежными системами — Phase 2

## 3. Посещаемость & учет занятий

### Статус реализации
- ❌ Полностью отсутствует

### Юзкейсы

#### 3.1 Регистрация посещения
**Актеры:** Тренер, рецепционист

⚠️ **ВАЖНО:** Используется **ОТЛОЖЕННОЕ СПИСАНИЕ** — баланс убывает не при CheckinClient, а при MarkInstanceAsCompleted (см. [`ATTENDANCE_DEDUCTION_PATTERNS.md`](./ATTENDANCE_DEDUCTION_PATTERNS.md))

**Кейсы:**
- UC-3.1.1: Отметить посещение клиента на занятии (быстрая операция)
  - Клиент выбирается из списка участников группы
  - Attendance создаётся с status = REGISTERED (черновик)
  - ✅ Membership.balance НЕ меняется в этот момент!
  - Баланс убудет ПОЗЖЕ (при MarkInstanceAsCompleted)
  
- UC-3.1.2: Отметить посещение на день-в-день (drop-in) без абонемента
  - Поиск клиента или гость
  - ⚠️ Списание может быть немедленное (требует уточнения)
  
- UC-3.1.3: Переместить (перенести) посещение на другое занятие
  - Клиент был зарегистрирован на Вторник, хочет на Четверг
  - Действие: RemoveAttendance (из Вторника) + CheckinClient (на Четверг)
  - Никакие балансы не откатываются (ещё не были списаны)
  
- UC-3.1.4: Отмена посещения
  - Если REGISTERED (до Complete): RemoveAttendance (удалить черновик)
  - Если ATTENDED (после Complete): CancelAttendance (откатить с восстановлением баланса)

**Статус:** Критично, нужно:
- CheckinClient (создать REGISTERED Attendance)
- RemoveAttendance (удалить черновик - без эффектов)
- CancelAttendance (отменить финализированное с восстановлением баланса)
- MarkInstanceAsCompleted (финализировать посещение + списать)
- TransferCheckin (переместить между занятиями)
- Модель Attendance с status = {REGISTERED, ATTENDED, CANCELLED}

#### 3.2 Управление посещаемостью группы
**Актеры:** Тренер, администратор

**Кейсы:**
- UC-3.2.1: Просмотр списка участников группы перед занятием (кто записан)
- UC-3.2.2: Отметить присутствие/отсутствие массово
- UC-3.2.3: Просмотр истории посещений конкретного клиента
- UC-3.2.4: Отчет по посещаемости группы за период (кто часто ходит, кто редко)
- UC-3.2.5: Уведомление тренеру об отсутствии постоянного участника

**Статус:** Нужно:
- GroupAttendanceList (список перед занятием)
- ClientAttendanceHistory
- GroupAttendanceReport (за период)
- Фоновое уведомление за отсутствие

#### 3.3 Отчеты по посещаемости
**Актеры:** Администратор, тренер

**Кейсы:**
- UC-3.3.1: Рейтинг активности клиентов (кто наиболее активен)
- UC-3.3.2: Анализ churn (кто перестал ходить)
- UC-3.3.3: Средняя загрузка группы (кол-во участников в среднем)
- UC-3.3.4: Данные о посещаемости для финансового планирования

**Статус:** Phase 2 (аналитика), но нужно спланировать хранение данных

## 4. Расписание занятий

### Статус реализации
- ⚠️ **Частично** — есть schedule_slots при создании группы, но нет управления расписанием

### Юзкейсы

#### 4.1 Управление расписанием группы
**Актеры:** Администратор, координатор

**Кейсы:**
- UC-4.1.1: Создание группы с расписанием (уже есть в CreateGroup)
- UC-4.1.2: Изменение расписания группы (добавить/удалить/изменить слот)
- UC-4.1.3: Просмотр расписания группы
- UC-4.1.4: Просмотр всех групп в виде календаря (что когда занимается)

**Статус:** Нужно расширить:
- UpdateGroupSchedule (добавить/удалить слоты)
- GroupScheduleDetail
- OrgScheduleCalendar (полное расписание организации)

#### 4.2 Планирование занятий (instance)
**Актеры:** Администратор, тренер

**Кейсы:**
- UC-4.2.1: Генерирование реальных занятий (instances) из расписания группы
  - Например, группа "Йога ПН/СР" каждый месяц генерирует 8 instances
  
- UC-4.2.2: Отмена отдельного занятия (например, выходной или болезнь тренера)
  - Все REGISTERED Attendance → CANCELLED (без списания баланса!)
  - GroupInstance → CANCELLED
  
- UC-4.2.3: Перенос занятия на другой день/время
  - Изменение GroupInstance.scheduled_at
  
- UC-4.2.4: Назначение тренера на занятие
  - GroupInstance.trainer_id = trainerId
  
- UC-4.2.5: Просмотр расписания тренера (его рабочий график)
  - Какие GroupInstance назначены этому тренеру
  
- **UC-4.2.6: Завершение занятия и финализация посещений** ⚠️ КРИТИЧНО
  - MarkInstanceAsCompleted(instanceId)
  - Это ГЛАВНЫЙ момент, когда:
    - GroupInstance.status = COMPLETED
    - Для каждого REGISTERED Attendance:
      - Membership.balance -= 1
      - Attendance.status = ATTENDED
    - Запускаются async события (уведомления, задачи)
  - 🟢 Ключевой UC для правильной работы системы!

**Статус:** Новое направление, нужно:
- GenerateGroupInstances (планирование)
- CancelInstance / RescheduleInstance
- AssignTrainerToInstance
- TrainerSchedule
- **MarkInstanceAsCompleted** ← КРИТИЧНО! (отложенное списание)

#### 4.3 Управление помещениями/залами
**Актеры:** Администратор

**Кейсы:**
- UC-4.3.1: Создание помещения/зала (название, емкость, оборудование)
- UC-4.3.2: Привязка группы к залу (где она проводится)
- UC-4.3.3: Просмотр занятости зала (конфликты расписания)
- UC-4.3.4: Уведомление при перекрытии (два группы в один зал в один час)

**Статус:** Новое, нужно:
- CreateRoom / UpdateRoom
- AssignGroupToRoom
- RoomAvailability (проверка конфликтов)

#### 4.4 Синхронизация и экспорт расписания
**Актеры:** Администратор, клиент

**Кейсы:**
- UC-4.4.1: Экспорт расписания в iCal (для Apple Calendar, Google Calendar)
- UC-4.4.2: Экспорт расписания в PDF для печати
- UC-4.4.3: Синхронизация расписания на мобильное приложение

**Статус:** Phase 2, но важно спланировать

## 📝 Дополнительные замечания

### Про размер (до 1000 клиентов)
- ✅ **Нет** нужды в кэшировании, сложной индексации, очередях
- ✅ **Нет** нужды в микросервисной архитектуре  
- ✅ **Есть** нужда в простоте и надежности
- ⚠️ **Нужно** учитывать: многопользовательский доступ (конфликты при одновременном редактировании)

### Data Model (примерно)
```
Client
├── profile (name, contacts, avatar)
├── balance (monetary balance journal)
├── memberships (active, inactive)
```

### Security & Permissions
- Все операции должны проверять orgId (мультитенантность)
- Тренер видит только своих клиентов и свои занятия
- Администратор видит всё
- Клиент видит только свои абонементы, расписание
