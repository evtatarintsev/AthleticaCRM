# Личный счёт клиента

## Обзор

Каждый клиент имеет личный счёт — баланс внутри организации. Баланс не пополняется напрямую: он изменяется исключительно как часть другой операции (продажа, возврат, административная корректировка). Это сохраняет связь каждого движения средств с конкретным событием и упрощает аудит.

Текущий баланс клиента вычисляется как `SUM(amount)` по всем записям журнала `client_balance_journal`.

---

## Юз-кейсы

### UC-1: Сдача при продаже → на счёт

Клиент платит больше, чем стоит товар или услуга. Разница зачисляется на его личный счёт вместо выдачи сдачи наличными.

**Пример:** абонемент стоит 1800 ₽, клиент даёт 2000 ₽ → 200 ₽ уходят на счёт.

- При оформлении продажи сотрудник указывает сумму, полученную от клиента.
- Если сумма больше цены, разница записывается в журнал с типом `sale_overpayment`.
- Операция атомарна с созданием самой продажи.

---

### UC-2: Частичная или полная оплата с баланса

При оформлении продажи сотрудник указывает, что часть суммы списывается с баланса клиента.

**Пример:** абонемент 1800 ₽, на счёте 200 ₽ → клиент доплачивает 1600 ₽, 200 ₽ списываются со счёта.

- Система проверяет, что списываемая сумма не превышает текущий баланс клиента.
- В журнал записывается операция с типом `sale_payment` и отрицательным `amount`.
- Операция атомарна с созданием продажи.
- Смешанная оплата (баланс + наличные/карта) — стандартный сценарий; оба движения средств фиксируются как части одной продажи.

---

### UC-3: Возврат продажи → средства на счёт

При отмене или возврате продажи оплаченная сумма возвращается на баланс клиента.

- Если оригинальная продажа была оплачена частично с баланса, возврат зачисляет обратно ровно ту часть, что была списана.
- В журнал записывается операция с типом `refund` и положительным `amount`.
- Операция атомарна с изменением статуса продажи.

---

### UC-4: Административная корректировка

Сотрудник с соответствующим правом может вручную пополнить или списать баланс клиента без привязки к продаже.

**Типичные случаи:**
- начисление бонуса или промо-кредита;
- исправление ошибки предыдущей операции;
- перенос средств между клиентами (например, члены одной семьи).

- Тип операции: `admin_credit` (пополнение) или `admin_debit` (списание).
- Поле `note` обязательно — причина корректировки должна быть зафиксирована.
- `performed_by` фиксирует, какой пользователь выполнил операцию.

---

### UC-5: Просмотр истории баланса

Сотрудник или администратор открывает карточку клиента и видит хронологическую ленту операций с балансом.

Каждая запись содержит:
- дату и время операции;
- тип (`sale_overpayment`, `sale_payment`, `refund`, `admin_credit`, `admin_debit`);
- сумму изменения (`amount`);
- баланс после операции (`balance_after`);
- ссылку на связанную сущность (продажу, возврат), если есть;
- комментарий (для административных операций);
- имя сотрудника, выполнившего операцию.

---

## Принятые решения

### Журнал вместо поля на клиенте

Баланс хранится не как одно поле `balance` в таблице `clients`, а как журнал записей в `client_balance_journal`. Текущий баланс = `SUM(amount)`.

**Почему:** полный аудит-трейл без дополнительных усилий, возможность восстановить баланс на любой момент времени, защита от "потерянных" изменений при конкурентных записях.

### Знаковый `amount` вместо пары `credit`/`debit`

Одно поле со знаком: положительное — пополнение, отрицательное — списание.

**Почему:** проще агрегировать (`SUM`), меньше ошибок при чтении. Тип операции (`operation_type`) задаёт смысловой контекст, знак задаёт направление.

### `balance_after` — снимок после каждой операции

Каждая запись хранит баланс клиента после операции.

**Почему:** позволяет отображать историю с бегущим остатком без рекурсивного суммирования, служит точкой проверки целостности данных.

### Полиморфная ссылка `ref_entity_type` + `ref_entity_id`

Связь с родительской сущностью (продажа, возврат и т.д.) хранится как пара строка + UUID без жёсткого FK.

**Почему:** не ограничивает схему будущими типами сущностей, не требует таблицы-справочника. Целостность обеспечивается на уровне приложения при создании записи.

### Нет прямого пополнения через интерфейс клиента

Клиент не может самостоятельно положить деньги на счёт. Баланс меняется только через операции (продажа, возврат, корректировка сотрудником).

**Почему:** исключает сценарий "деньги лежат на счёте без привязки к услуге", упрощает финансовую отчётность, снижает риск злоупотреблений.

### Обязательный `note` для административных операций

Проверяется на уровне приложения, не через DB constraint.

**Почему:** PostgreSQL не поддерживает условный NOT NULL через ENUM нативно без громоздких триггеров. Бизнес-логика в приложении проще и тестируется стандартными юнит-тестами.

## Схема БД

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID | Первичный ключ (uuidv7) |
| `org_id` | UUID | Организация |
| `client_id` | UUID | Клиент |
| `amount` | NUMERIC(10,2) | Изменение баланса (+ пополнение, − списание) |
| `balance_after` | NUMERIC(10,2) | Баланс клиента после операции |
| `operation_type` | ENUM | Тип операции |
| `ref_entity_type` | VARCHAR | Тип связанной сущности (`sale`, `refund`, …) |
| `ref_entity_id` | UUID | ID связанной сущности |
| `note` | TEXT | Комментарий (обязателен для admin_credit / admin_debit) |
| `performed_by` | UUID | Пользователь, выполнивший операцию |
| `created_at` | TIMESTAMPTZ | Время операции |

Типы `operation_type`:

| Значение | Направление | Описание |
|----------|-------------|----------|
| `sale_overpayment` | + | Сдача при продаже |
| `sale_payment` | − | Оплата продажи с баланса |
| `refund` | + | Возврат при отмене продажи |
| `admin_credit` | + | Административное пополнение |
| `admin_debit` | − | Административное списание |
